# SmartClaw 开发计划（TDD 版）

## 开发文档更新指南

### 开发流程

1. **阅读当前进度** - 查看本文档的「模块进度汇总」，确认当前所在模块和任务
2. **遵循模块顺序** - 按 A → B → C → D → E → F → G → H 顺序开发，不跳模块
3. **TDD 开发循环** - 每个任务遵循 Red-Green-Refactor 循环
4. **完成标记** - 任务完成后立即更新本文档

### 更新规则

| 操作 | 更新内容 |
|-----|---------|
| 开始任务 | 将 `[ ]` 改为 `[*]` |
| 完成任务 | 将 `[*]` 改为 `[x]`，填写「任务完成日期」和「备注」 |
| 遇到阻塞 | 将 `[ ]` 改为 `[!]`，在「备注」说明阻塞原因 |
| 模块完成 | 更新「模块进度汇总」中的已完成任务数 |

### 更新步骤（每个任务完成后）

1. 在对应任务行：
   - 将 `开发进度` 列的 `[ ]` 或 `[*]` 改为 `[x]`
   - 在 `任务完成日期` 列填写日期（格式：YYYY-MM-DD）
   - 在 `备注` 列填写实际开发情况（遇到的问题、解决方案等）

2. 在「模块进度汇总」中：
   - 将对应模块的「已完成」数 +1
   - 将「未完成」数 -1
   - 更新「完成率」百分比

3. 提交 Git：
   ```bash
   git add Develop_schedule.md
   git commit -m "docs: 完成任务 {任务编号} - {任务名称}"
   ```

### 日期格式规范

- 任务完成日期：`YYYY-MM-DD`（如 2024-01-15）
- 备注中的时间戳：`HH:MM`（如 14:30）

### 备注填写建议

| 情况 | 备注示例 |
|-----|---------|
| 正常完成 | `按计划完成` |
| 有变更 | `原设计X改为Y，原因：...` |
| 遇到问题 | `问题：...；解决：...` |
| 依赖阻塞 | `阻塞：等待Z模块完成` |

### 验收规则（每个任务必须满足）

完成任何开发项时，必须确认以下验收规则：

| 类别 | 规则 | 验证方式 |
|-----|------|---------|
| 测试 | 测试用例覆盖正常场景 | pytest 通过 |
| 测试 | 测试用例覆盖边界场景 | pytest 通过 |
| 代码 | 符合 PEP 8 规范 | ruff 无警告 |
| 代码 | 类型注解完整 | mypy 通过 |
| 代码 | 文档字符串完整 | 代码审查 |
| 功能 | 符合 DEV_SPEC 规范 | 对照规范文档 |
| 功能 | 错误处理正确 | 异常测试通过 |
| 功能 | 日志记录完整 | 日志输出验证 |

## 格式说明

### 开发进度标记
- `[ ]` 未开始
- `[*]` 进行中
- `[x]` 已完成
- `[!]` 阻塞中

### TDD 开发原则
每个开发项遵循 **Red-Green-Refactor** 循环：
1. **Red** - 先编写失败的测试用例
2. **Green** - 编写最少代码使测试通过
3. **Refactor** - 重构优化，确保测试仍然通过
